Анализ проекта ESP32 MotionController — подводные камни
========================================================

1. Потокобезопасность (самое критичное)
---------------------------------------
Проект работает в трёх контекстах одновременно: main loop (core 1),
LCD task (core 0), аппаратный таймер (ISR). Синхронизация между ними
отсутствует полностью.

include/state.h — класс State<T> не имеет ни мьютексов, ни атомарных
операций. get() и set() вызываются из разных ядер одновременно.
Для float значений (vibroAngle, vibroFr) это означает torn reads —
чтение наполовину записанного значения.

src/global.cpp:20 — bool vibroDir не атомарный, читается и пишется
из main loop, потенциально читается из LCD task.

src/global.cpp:23 — объект stepper (GStepper2) используется из трёх
контекстов:
  - ISR: stepper.tick() (main.cpp:53)
  - main loop: stepper.setSpeed(), stepper.setTarget() (main.cpp:152, 163)
  - observer callbacks: stepper.setMaxSpeed(), stepper.enable() (src/observer/)

Это прямой путь к пропуску шагов и порче внутреннего состояния степпера.

Рекомендация: добавить portMUX_TYPE spinlock в State<T> для get()/set(),
сделать vibroDir атомарным (std::atomic<bool>), защитить доступ к stepper
мьютексом или перенести всё управление на одно ядро.


2. Гонка при инициализации
--------------------------
src/lcd/lcd.cpp — LCD task создаётся через xTaskCreatePinnedToCore
в lcdInit(), который вызывается в main.cpp:62. Но observerAll()
вызывается позже на строке 82, а init() для всех State переменных —
на строках 84-92. LCD task уже может работать и читать
неинициализированные значения.

Рекомендация: использовать семафор или флаг готовности, чтобы LCD task
ждал завершения setup().


3. Observer pattern — каскадные вызовы
--------------------------------------
src/observer/tmcDriverMicrostepObserver.cpp:14-15 — при изменении
микрошага вызываются vibroAngle.notifyObservers() и
vibroFr.notifyObservers(). Это каскадный вызов observer из observer.
Если в будущем добавится обратная зависимость — получите бесконечную
рекурсию. Также std::vector<std::function> в notifyObservers()
не защищён от модификации во время итерации.


4. Null pointer dereference в UI
---------------------------------
Все файлы рендеринга UI элементов (src/ui/ItemSwitch.cpp,
ItemEDITINT.cpp, ItemEditFloat.cpp, ItemMICROSTEP.cpp, ItemButton.cpp)
обращаются к item->value->get() без проверки на nullptr.
В ui.h:34-35 указатели value и valueF инициализированы как nullptr.
Если элемент меню сконфигурирован неправильно — краш.


5. Переполнение в ItemMICROSTEP
--------------------------------
src/ui/ItemMICROSTEP.cpp:14 — uint16_t t = item->value->get() * 2.
Если текущее значение 256, результат 512 — это ок. Но если значение
уже на максимуме и умножается до проверки — промежуточный результат
может переполнить uint16_t (65535). Проверка if (t >= item->max)
идёт после арифметики.


6. EEPROM персистентность отключена
------------------------------------
include/state.h:64-98 — методы save() и load() полностью
закомментированы. Все настройки (ток, микрошаг, скорость) теряются
при перезагрузке. Адреса в addresses.h определены, но не используются.


7. Неинициализированные цвета в UI
-----------------------------------
src/ui/ui.h:52-54 — поля colorActive, colorInactive, colorBg
в структуре itemAction объявлены без значений по умолчанию.
В src/screen/addMenuElement.cpp при создании элементов эти поля
не всегда заполняются — на экране будут мусорные цвета.


8. Hardcoded скорость в continuous mode
---------------------------------------
src/main.cpp:163 — stepper.setSpeed(1000) захардкожено. Значение
constRps и tmcStepperMaxSpeed игнорируются в continuous mode.
Пользователь меняет скорость в меню, но мотор всегда крутится на 1000.


9. TMC2160 неполная инициализация
---------------------------------
src/tmc/tmcInit.cpp — вызываются только driver.begin() и
driver.toff(5). Ток, микрошаг, chopper mode не устанавливаются
при старте. Они применяются только через observer при вызове init(),
но если observer ещё не зарегистрирован (см. пункт 2) — драйвер
работает с дефолтными значениями чипа.


10. Мёртвый код
---------------
- src/main.cpp:218-240 — TaskDbLoop и initTaskDb() полностью
  закомментированы внутри, но функция вызывается в setup.
- src/main.cpp:35 — #define FREQUENCY 100000 не используется,
  таймер инициализируется с timerBegin(10000).
- Большой блок закомментированного кода в loop() (строки 170-213).
